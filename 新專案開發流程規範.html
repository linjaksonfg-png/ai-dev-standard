二日常開發流程

工程師從 Issue Board 領任務
先確認 Issue 已掛到正確 Milestone，再開 feature/issue-編號-簡述 branch 開發
Push → 自動建立 PR
PR 必須能對應 Issue / Milestone，且不能脫離該批交付範圍
🤖 第一關：自動化檢查
✅ CI（lint + test + build）
✅ AI Code Review（風格、安全性、變更範圍）
✅ 自動部署 Preview 環境
❌ 未通過 → 打回工程師修改
👔 第二關：老闆審批
👁️ 查看 Preview 畫面
📄 閱讀 AI Review 摘要
✅ 確認功能符合需求
🚀 老闆 Approve → 合併 PR → 自動部署 Production

三各階段詳細說明

階段 1：開 Repo + 初始化

步驟 說明
建立 Git Repo GitHub / GitLab / Gitea 皆可
設定 branch 保護 main 禁止直接 push，只接受 PR 合併
設定 PR 規則 至少 1 人 approve（老闆）+ CI 必須通過
建立 Milestone 策略 把 GitHub Milestone 當成交付批次，而不是分類標籤；凡是非 trivial 工作原則上都要掛 milestone
建立 Issue / PR 模板 模板必須要求填寫 milestone、驗證方式、風險與回退方案
設定 CODEOWNERS * @老闆帳號，所有 PR 自動指派老闆 review
初始化專案結構 複製 AI 規則檔（AGENTS.md, CLAUDE.md, GEMINI.md）
初始化 .ai-memory/ 建立記憶目錄結構（見 AGENTS.md） 
# CODEOWNERS 檔案內容（放在 repo 根目錄）
* @老闆的GitHub帳號

階段 2：建立開發端

步驟 說明
各工程師 clone repo 各自本地開發，互不干擾
安裝依賴 npm install / pip install 等
確認本地 dev server 確保每個人都能在本地看到程式結果
首次 push 推到 develop 分支作為開發基準

階段 3：建立 CI/CD Pipeline

CI（持續整合）— Repo 層級設定

每個 PR 建立時自動執行：

# .github/workflows/ci.yml 範例
name: CI
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 安裝依賴
        run: npm install
      - name: 程式碼檢查
        run: npm run lint
      - name: 單元測試
        run: npm test
      - name: 建置確認
        run: npm run build

AI Code Review — 自動化第一關

工具選項 說明
CodeRabbit SaaS，接 GitHub 即用，自動寫 review comments
GitHub Copilot Review GitHub 原生，需 Copilot 授權
PR-Agent (Codium) 開源可自建，支援摘要 + 建議 + 自動標籤

AI Review 負責檢查：

  • ✅ 程式碼風格與命名規範
  • ✅ 基本錯誤（空 catch、未使用變數）
  • ✅ 安全性（硬編碼密碼、注入風險）
  • ✅ 變更範圍（是否改了不該改的檔案）
  • ✅ 產生 AI 摘要（方便老闆快速了解改了什麼）

CD — Preview 部署

PR #42 → https://pr-42.preview.example.com
PR #43 → https://pr-43.preview.example.com
每個 PR 獨立 preview URL，避免多人 PR 互相覆蓋。

階段 4：建立伺服端

環境 對應分支 用途
Preview 各 PR 分支 合併前預覽，每個 PR 獨立 URL
Production main 正式環境，合併後自動部署

部署方式選項：

方案 適用場景
Vercel / Netlify 前端專案，自動 preview + production
Docker + Webhook 全端專案，伺服器收到 webhook 自動拉取部署
GitHub Actions + SSH 自建環境，CI 跑完後 SSH 到伺服器部署

階段 5：驗收完整流程

  • 工程師開 feature branch，push 成功
  • PR 自動建立，CI 自動開始跑
  • AI Review 自動留下 review comments
  • Preview 環境自動部署，URL 可訪問
  • 老闆收到通知，可查看 preview + AI 摘要
  • 老闆 approve 後，PR 可合併
  • 合併後 production 自動部署
  • Production 環境功能正常

四分支策略

main
正式環境
↑ PR 合併（老闆 approve）
develop
開發整合
↑ PR 合併（CI 通過即可）
feature/issue-42-login
功能分支
feature/issue-43-payment
功能分支
hotfix/fix-critical-bug
緊急修復
分支類型 命名規範 從哪開 合到哪
feature feature/issue-編號-簡述 develop develop → main
hotfix hotfix/簡述 main main + develop
Issue 是正式工作單位，Milestone 是交付批次，PR 是審查與實作載體。labels 只負責分類，不取代 milestone。

五衝突預防

規則 說明
📅 每天 rebase 工程師每天從 develop rebase 最新程式碼
📢 大改動公告 重構或架構變更前，在群組公告
📦 模組分工 盡量避免兩人同時改同一模組
📏 小 PR 原則 每個 PR ≤ 300 行變更，方便 review

六通知機制

讓老闆能快速審核，不成為瓶頸
事件 通知方式 內容
PR 建立 LINE / Slack / Email PR 連結 + AI 摘要
CI 通過 自動標記 ✅ Preview URL
CI 失敗 通知工程師 錯誤日誌連結
等待 approve 超過 2h 再次提醒老闆 —
